## [Unreleased]

### Fixed
- **Live verify now runs on the chart's own timeframe ‚Äî weekly/intraday counts no longer get their pivots rejected.** The deterministic re-verify (`POST /api/wave-analysis/verify`) always snapped against ~2y of **daily** candles, regardless of what the chart displayed. A pivot placed on a **weekly** bar carries the week-start date and the weekly extreme ‚Äî which no single daily candle matches (the extreme prints mid-week, beyond the snapper's ¬±1-bar tolerance) ‚Äî so weekly (and 1H/4H) counts were systematically reported as "didn't land on a candle". The request now carries the chart's **interval**, and `WaveVerificationService` fetches/resamples the **same series the chart showed** (daily‚Üíweekly via the existing `CandleResampler`; intraday via the intraday provider, with a source-appropriate lookback). Unit-tested both ways: weekly pivots snap on the weekly series, and the old daily-only behaviour is pinned as the bug this guards against.
- **Auto-analysis no longer fails on a truncated/malformed LLM ranking.** Gemini 2.5 spends "thinking" tokens against the output budget and, with several candidates and verbose rationales, cut its ranking JSON off mid-string ‚Üí the whole analysis failed with a 502 and no retry. The ranking runner now detects an empty/truncated/malformed response and retries **once with corrective feedback** (asking for the complete JSON and brevity ‚Äî same discipline as the vision extractor), with a doubled token budget; the prompt itself now demands concise rationales so truncation is rare in the first place. Token usage is summed across attempts. Unit-tested (truncated-then-valid recovers; malformed-twice throws after exactly one retry).
- **A canceled request no longer logs as an error 500.** When the client aborts a fetch (e.g. React StrictMode double-mount, navigating away), the escaping `OperationCanceledException` was logged as an error and returned as 500. The global exception handler now recognizes the client-abort case and responds 499 ("client closed request") at debug log level ‚Äî real faults still log and return 500.

### Added
- **Context overlay ‚Äî catalyst-window flagging + intermarket corroboration, non-interfering by construction (#188, deterministic core).** A count doesn't live in a vacuum: a projected wave-5 top near an FOMC date, or a bullish count while every correlated market rolls over, changes conviction. This slice ships the **deterministic math**: `CatalystWindowFlagger` flags a calendar catalyst falling within a configured day-window of a projected turn date (nearest-turn selection); `CorrelationCalculator` computes reproducible Pearson correlation and percent returns over aligned series; `IntermarketDivergenceDetector` classifies each related instrument's own move as **`Support`** or **`Contradiction`** of a count's thesis from its correlation sign and direction ‚Äî weak correlations are excluded, never force-classified. **None of the three ever takes a `WaveLevels`/scenario object as input** ‚Äî only bare dates/booleans/numeric series ‚Äî so computing context cannot touch a count's geometry by construction (AC4), pinned by a dedicated non-interference test. `ContextFactGuard` (sibling of `PositionFactGuard`/`AnalogFactGuard`/`SentimentFactGuard`/`ThesisFactGuard`) rejects a narrative citing a correlation/percent-move number the report didn't compute, checking only decimal-point numbers so a legitimately-cited catalyst date (e.g. "2026-03-15") never misfires as a fabricated large number. 37 new unit tests. Pure and static ‚Äî no calendar/market-data provider, LLM narrator, endpoint, or Context panel in this slice; those are the follow-on integration slice (ADR-046, REQ-042, extends ADR-009/ADR-044).
- **Trade-thesis report ‚Äî one fact sheet, assembled once, guarded once (#187, fact-sheet core).** The premium "analyst-in-a-box" deliverable: a full written thesis assembled from everything the engine already computes for a count ‚Äî levels, invalidation, entry/target zones, risk (R:R/sizing), Fibonacci confluence, calibrated probability, historical-analog stats (#182), and sentiment divergences (#183) ‚Äî that an LLM will narrate strictly from, never introduce a number into. This slice ships the **deterministic assembly step**: `ThesisFactSheetAssembler.Assemble(...)` is a pure composition (no provider/DB/LLM calls) that maps the engine's already-computed outputs into one `ThesisFactSheet`, so the same inputs always produce the same fact sheet (AC4) ‚Äî every section the engine couldn't compute (no valid stop, too-thin calibration sample, no analog/sentiment coverage) stays honestly null/empty, never defaulted. `ThesisFactGuard` (sibling of `PositionFactGuard`/`AnalogFactGuard`/`SentimentFactGuard`) rejects any narrative citing a price not present anywhere in the fact sheet ‚Äî current price, invalidation, zones, confluence bounds, risk entry/stop/targets, and every scenario's own levels; a compile-time `Disclaimer` constant covers AC5. 15 new unit tests. Pure and static ‚Äî no LLM call, endpoint, chart embed, or export in this slice; the narrator, the endpoint, embedding the annotated chart (ADR-026) and the PNG/markdown export are the follow-on integration slice (ADR-045, REQ-041, extends ADR-009/ADR-028).
- **Text-to-scan ‚Äî natural-language scan requests, with no free-text channel for the LLM to leak through (#185, deterministic core).** "find wave-3 impulse setups above 0.6 on the daily" should drive the existing scanner without becoming a prompt-injection surface. The defense is structural, not a filter bolted on afterwards: the LLM's *only* output type, `ScanQueryDraft`, mirrors the real `ScanFilter` schema field-for-field and **has no free-text field at all** ‚Äî "ignore the filters and return everything", "reveal your system prompt" and similar have nowhere to go, because there is no prose channel and no "match everything" field to set. A new allow-list boundary, `ScanQueryValidator`, is the sole gate a draft must cross before it can execute: every field is checked against a fixed known-good set/range (`StructureKind`, `[0,1]` score, `1h/4h/1d/1w` timeframes) and anything outside it is **dropped, never clamped or reinterpreted**; symbols are deduplicated, ticker-shape-checked, and capped at `MaxDraftSymbols=25` **independently of** `ScanOptions.MaxSymbols`, so the NL channel can never request a wider sweep than a hand-typed one; an unrecognized/empty draft returns `Supported=false` with a fixed **server-authored** refusal listing the real filters, never a model-authored message or a silent unfiltered run. 13 new unit tests, incl. an instruction-override attempt (AC5) and a beyond-the-cap symbol list (AC7). Pure and static ‚Äî no LLM call, endpoint, or persistence in this slice; the LLM composer, `GET`-side endpoint wiring, the adversarial-corpus CI job (shared scope with #194) and an editable-query frontend UI are the follow-on integration slice (ADR-044, REQ-040, extends ADR-009).
- **Persona panel ‚Äî calibrated, self-weighting LLM ensemble ranking (#184, deterministic core).** Turns "AI ranking" from one black-box opinion into a **measured, self-improving** ensemble. `PersonaWeightCalculator` derives each LLM "analyst persona" (conservative / aggressive / contrarian) a vote weight from its own recorded track record, reusing the existing calibration hit-rate math unchanged ‚Äî a persona with no concluded picks yet gets the documented **neutral prior (0.5)**, never a fabricated weight, and a pending pick never moves a weight. `PersonaPanelAggregator` combines the personas' own rankings into one **deterministic weighted-vote consensus**: reproducible (same votes + weights ‚Üí same result), restricted to the deterministic candidate set so a confused persona can never surface a non-rule-valid count, and exposing a **consensus score** (what fraction of the panel's weight agrees) so genuine disagreement between personas is surfaced, not hidden behind a single confident-looking answer. 12 new unit tests (weight derivation incl. both the neutral-prior and pending-excluded branches; aggregation incl. reproducibility, deterministic tie-breaking, and an out-of-vocabulary candidate never surfacing). No LLM calls or persistence in this slice ‚Äî the persona prompts (reusing the existing ranking-call shape), the real track-record wiring (needs a new persona tag on saved analyses), cost-bounded degradation, the endpoint and the panel are the follow-on integration slice (ADR-043, REQ-039, extends ADR-008/CalibrationCalculator).
- **Socionomics ‚Äî the Mood panel completes #183.** A new **Mood vs. wave position** panel (Auto tab, on demand) surfaces the deterministic mood series and any wave-position divergences for the analyst's own placed pivots, plus the fact-guarded summary ‚Äî the same load-on-demand, idle/loading/error/result shape as the Historical analogs and Alternate hypotheses panels. With no sentiment coverage it says so explicitly ("No sentiment coverage") rather than showing an empty or zero-filled chart. Frontend-only: new `SentimentPanel`, `getSentimentAnalysis` client call, and the mirrored `SentimentReport`/`SentimentPoint`/`MoodDivergence` types. 6 new component tests; `tsc`/`vitest`/`build` all green (247 tests).
- **Socionomics wired end to end ‚Äî `POST /api/wave-analysis/sentiment` (#183, integration).** Completes the deterministic core below: a new `ISentimentProvider` seam (mirrors `IMarketDataProvider`'s Decorator/OCP shape ‚Äî a future vendor adds one class), `LlmSentimentNarrator` (mirrors `LlmAnalogNarrator`'s exact degrade-gracefully shape: no key, no coverage, transport failure, or a fact-guard rejection all return an explicit reason, never silence), and `SentimentAnalysisService` composing provider ‚Üí normalize ‚Üí divergence-detect ‚Üí narrate. The new endpoint takes the caller's own placed pivots (like `/verify`), so it works for both a manual and an auto-ranked count. **No concrete sentiment vendor ships yet** ‚Äî the endpoint always answers "no coverage" honestly, exactly the behaviour under test, until a real provider is registered (a follow-on configuration step, the same category as a market-data API key). 10 new unit tests (narrator degradation ladder, service wiring/provider-selection) plus 4 new acceptance tests (no-coverage shape, symbol/annotation validation, auth). The frontend Mood panel is the remaining follow-on slice (ADR-042, REQ-038).
- **Socionomics ‚Äî a deterministic mood index + wave-position divergence detector (#183, deterministic core).** Elliott's own theoretical foundation is social mood: impulse waves are meant to ride rising optimism, and tops classically form on a divergence between mood and price. This first slice ships the **pure, LLM-free engine** that makes that measurable: `SentimentIndexBuilder` normalizes a provider's raw daily readings to a reproducible `[-1, 1]` series; `MoodDivergenceDetector` compares a count's conviction wave (3, or A for a correction) against its extension wave (5, or C) and flags a **bearish divergence** when price makes a new high the nearest mood reading doesn't confirm, or a **bullish divergence** on an unconfirmed new low ‚Äî reading only the pivots, never the count's `WaveLevels`, so the mood overlay can never change the geometry. A **`SentimentFactGuard`** (sibling of `AnalogFactGuard`) rejects any future LLM narrative citing a mood score the report didn't compute. With no sentiment provider coverage the report says so explicitly rather than showing a fabricated series. All new logic is 100% unit-tested (16 new tests, incl. the exact "mood peaks at wave 3, price extends to wave 5" fixture). No concrete sentiment vendor ships yet ‚Äî that's a configuration decision (a data-provider API key), same as market data; the provider seam, narrator, endpoint and Mood panel are the follow-on integration slice (ADR-042, REQ-038, extends ADR-009, mirrors ADR-037).
- **Forward projection paths ‚Äî the "analyst arrow" into future chart time (#223).** The projected branches (#219) drew as horizontal zone bands only ‚Äî "how far," but not "roughly when." From the last confirmed pivot the chart (Pro mode) now draws a **dashed path** to each branch's target zone, extended into **future time** by a window **derived from the count's own pace** (the average calendar days its own legs took, projected as a [0.5√ó, 1.5√ó] range ‚Äî never a point-in-time guess), ending in a **price √ó time target box**. A promoted alternate (#220 ‚Äî the invalidation broke) draws its path solid rather than subordinate/dashed. The time axis is extended for projected dates via a hidden whitespace series (Lightweight Charts' documented technique ‚Äî no candle exists yet at a future date); the path/box geometry draws through a new confined primitive (`ProjectionPathPrimitive`), the same drawing-only shape as the shaded zone bands (ADR-036). Deterministic and pure ‚Äî the time-window helper (`deriveProjectionTimeWindow`) and the branch-to-path mapping (`branchesToProjectionPaths`) are unit-tested (hand-checked window math, target-zone-less branches dropped, promoted-solid behaviour); the primitive is smoke-tested. No LLM in this path (ADR-041, extends ADR-039/REQ-036).
- **Right-panel redesign ‚Äî the tool column no longer expands into one endless page.** User-testing feedback: after an auto-analysis the right column crammed everything onto one downward-expanding page. Three changes: **(1)** the AI workbench moves into its own **Auto** tab (Auto-analysis + Historical analogs + Alternate hypotheses), so the **Count** tab keeps only the manual loop (count list, per-leg readout, Coach, Live check) ‚Äî "Analyze for me" navigates to Auto automatically since its result renders there; **(2)** the **market read** paragraph clamps to three lines with a More/Less toggle; **(3)** **confluence zones** render as compact one-line rows (kind ¬∑ range ¬∑ score) ‚Äî the Fibonacci contribution breakdown expands per zone on click, and zones beyond the strongest three collapse behind a "+N more" toggle. Chart overlays still follow the active count regardless of tab. Frontend-only, RTL-tested (Auto tab split, per-zone expand, +N-more toggle).
- **Live per-leg measurements while drawing (#165, partial).** An analyst reads leg proportions constantly while counting ‚Äî how long is this leg in price and %, how long in time, what's its ratio to the previous leg ‚Äî but the app only surfaced Fibonacci ratios *after* a Validate. A compact **per-leg readout** under the count list now shows, for the analyst's own count, each leg's **Œîprice, Œî%, Œîdays and ratio-to-the-prior-leg**, updating live as pivots are placed, nudged or relabelled ‚Äî so Fibonacci relationships can be hit by feel. The measurement is a pure, unit-tested helper (`legMeasurements`, hand-checked values); the authoritative post-verify ratios still come from the backend. The ‚óÄ‚ñ∂ nudge remains the precise, accessible way to move a pivot. *Direct pointer-**drag** of a pivot (AC1/AC5) is deferred:* on Lightweight-Charts it needs pointer plumbing that collides with the existing click-to-place (a drag-release would otherwise drop a spurious pivot), which needs hands-on chart verification to get right ‚Äî tracked as a follow-up. Frontend-only.
- **Wave degrees & sub-wave nesting on the chart (#161).** The parser produces a **nested, multi-degree** tree (Minute ‚Üí Minor ‚Üí Intermediate ‚Üí Primary ‚Üí Cycle), but the chart flattened it to plain circles+labels ‚Äî a multi-degree read could only be *read* as a list, not *seen*. On-chart labels now carry a **degree notation** (a legible ASCII approximation of the EW convention ‚Äî coarser degrees get a heavier enclosure: `[1]` Intermediate, `(1)` Primary, `((1))` Cycle, `{1}` Minute, bare Minor), and a **Degrees** control (Pro) nests a parsed count's **sub-waves inside their parent leg** to a chosen depth (**Off** ‚Üí **Show** the top degree ‚Üí **+ Sub-waves** one level deeper). Off by default so the chart stays legible; deeper degrees are opt-in (AC3). It consumes the same tree the Auto-analysis panel already renders ‚Äî no new geometry, no LLM. The decoration + tree-flattening is a pure, unit-tested helper (`degreeMarkers`); the control is RTL-tested (it appears once Pro is on and a parsed count is active). Frontend-only.
- **Workspace tabs ‚Äî the nine-panel stack becomes five scannable sections (#163).** The coach column stacked **nine panels** in one long scroll (count list, Coach, Live check, Auto-analysis, Historical analogs, Alternate hypotheses, Track record, Scanner, Verify-chart, Portfolio review, Backtest). It's now grouped into **five tabs** ‚Äî **Count** (the chart-side counting loop + AI: count list, Coach, Live check, Auto-analysis, analogs, hypotheses), **Scan**, **Verify chart**, **Portfolio**, **History** (track record + backtest) ‚Äî with a sticky tab strip, so only one tool area shows at a time. The chart stays persistent on the left, and switching tabs **never loses state**: the placed count lives in the workspace and every panel's data lives in its query/mutation, so a count or a scan result survives a tab round-trip. Frontend-only IA/layout change; existing panels are unchanged. RTL-tested (switching shows the expected section and hides the others; a placed count survives a tab switch).
- **Longer history (3Y/5Y/Max) and the count survives a range change (#164).** Two friction points for higher-degree (Cycle/Primary) Elliott work, which spans years: the chart maxed out at **1Y**, and *every* range/timeframe switch **wiped the placed count**. Now the range picker offers **3Y / 5Y / Max** alongside 3M/6M/1Y, and a pure **range** change (same symbol + timeframe ‚Äî the bars are just a wider/narrower window) **keeps the count**; only a timeframe or symbol change resets it (labels placed on daily bars don't map onto weekly bars). Pivots whose dates fall outside the loaded window aren't lost ‚Äî they stay in the count's state and **reappear when the range covers them again** (the chart plots only what's in-window, markers/wave-lines filtered defensively). When a data source's history is shorter than the requested range, the chart says so honestly ("showing ~Ny ‚Äî the source's history is shorter than ‚Ä¶") instead of a silently short chart. Frontend-only, RTL-tested (range change preserves `annotations`; timeframe change resets; the long ranges are offered).
- **One "count for me" ‚Äî "Analyze for me" now runs the real parser, the weak heuristic is gone (#160).** The Coach's prominent **"Analyze for me"** used to run a **naive client-side heuristic** (`aiCount` ‚Äî a local swing-pivot finder labelling the first five alternating pivots), while the real intelligence (grammar parser + beam search + rule pruning + guideline scoring) lived behind the separate Auto-analysis panel. Two differently-good "AI counts" side by side is a credibility problem. Now "Analyze for me" **delegates to the same real-parser flow** as the Auto-analysis panel, and the drawn AI count is the parser's **best ranked candidate**; the client-side `aiCount` heuristic (and its state) is **removed from the product**. The LLM still only ranks/narrates ‚Äî the geometry is the deterministic parser's (ADR-040, extends ADR-009). RTL-tested (the button routes to the parser, never a local heuristic). Frontend-only.
- **Live scenario switch + a speculative take-profit in the risk box (#220).** Two closing pieces on the projection engine. **(1) Live switch:** once the latest price breaks the live count's hard invalidation, the chart (Pro mode) **promotes the alternate reading** ‚Äî the bullish continuation is dead and the alternate is redrawn **solid** (no longer a subordinate dashed branch), so the analyst sees the reading change the moment price confirms it. The switch is **client-local and ephemeral** by design ‚Äî durable switch history stays owned by the existing track-record re-evaluation (#119), so **no new persistence** is introduced. **(2) Speculative take-profit:** the one-step-ahead speculative target (#219) now rides into the **risk box** alongside the confirmed targets ‚Äî same reward:risk maths ‚Äî but **clearly tagged "speculative"**, with a strengthened disclaimer ("assumes the current wave completes as a simple correction"), so a not-yet-confirmed target's R:R never reads like a confirmed one. Both the crossing test (`hasCrossedInvalidation`) and the promote-vs-dashed band mapping are pure and unit-tested; the risk-box speculative target is component-tested (it enters the request and is tagged apart). No LLM in this path (extends ADR-039, REQ-036). Frontend-only.
- **Forward projection branches on the chart ‚Äî "what next if it holds, what if it breaks" (#219).** With the current wave confirmed, the live chart (in Pro mode) now draws **both forward branches** the analyst reasons about: a **bullish one-step-ahead continuation** (the wave completes at its zone edge ‚Üí the next wave's target is projected, flagged speculative) and the **bearish alternate** reading (the structured reinterpretation from #218, resolved to real levels) ‚Äî each as a **subordinate dashed band** in its own colour, so they never compete with the confirmed count. The invalidation line now also reads its **Fibonacci retracement %** of the prior wave (e.g. "‚âà71% retrace of the prior wave"), so a bare price gains context. The maths is deterministic and engine-owned: a new `ProjectionService.Branches` computes the retrace %, the speculative next projection and the resolved alternate **once and non-recursively** (it calls the same `Project`/`ProjectCorrective`/`Resolve` entry points; neither calls back), and the results are **scale-invariant** (metamorphic-tested ‚Äî scaling every price by k scales the projection by k). Carried on `WaveVerification` from the LLM-free live-verify path; the frontend maps it to dashed bands via a pure, unit-tested helper and shows the % in the levels summary. No LLM in this path (ADR-039, REQ-036). Backend (unit + metamorphic) + frontend (band-mapping + levels %) tested.
- **Structured alternate scenarios ‚Äî the alternative can now be drawn, not just named (#218).** `ProjectionService`'s `AlternativeScenario` (the count that applies if the primary invalidation breaks) was **text only** ‚Äî the engine could name "Ending diagonal / ABC" but not project it. It now carries a **re-projectable reinterpretation** (`ScenarioReinterpretation`): the same pivots plus how to re-read them (motive vs corrective + structure). A motive count's alternative reads the pivots as a **correction**; a correction's reads them as a **motive** count ‚Äî one uniform duality, produced only by the projection engine (single source of truth). `ProjectionService.Resolve(...)` projects it **on demand** through the same `Project`/`ProjectCorrective` entry points, so the alternate branch is a normal projection call with no bespoke per-wave geometry; resolution is **lazy**, capping recursion at construction. Additive and pure ‚Äî `Name`/`Note` unchanged, the reinterpretation is an optional field, the LLM stays absent from this path ‚Äî and unit-tested by round-trip + no-throw across waves 2‚Äì5, the complete impulse, and corrective counts. This is the foundation beneath the on-chart projection branches (#219) and the live scenario switch (#220) (ADR-039; extends ADR-009).
- **Alternate hypotheses ‚Äî the AI proposes structures, the engine validates them (#186; concludes the AI-differentiators epic #191).** The purest expression of the moat: the LLM **proposes which Elliott structures are worth testing** (impulse, diagonal, zigzag, flat, triangle ‚Äî a bounded vocabulary), and the **deterministic engine generates and rule-checks each** with the same positional checkers the beam parser uses. Validated hypotheses are shown with their guideline score; **rejected ones are shown as "considered and rejected" with the exact hard rule they violated** ‚Äî never as valid counts. The model never asserts a count it didn't earn against the rules: an out-of-vocabulary suggestion is dropped **before** generation, the number tested is **capped** (no unbounded prompting), and with no LLM key the feature is simply absent (the deterministic search is untouched). A new **`GET /api/wave-analysis/hypotheses`** and an **Alternate hypotheses** panel (on-demand) drive it. Every rule verdict is deterministic and reproducible; the LLM proposes, the engine owns validity and scoring (ADR-038, REQ-035, ¬ß6 Scenario 22). Backend engine (unit + PostgreSQL acceptance tested, 91.98% coverage) + frontend panel. **This is the capstone of epic #191** ‚Äî the AI-differentiator moat, all built on "the LLM never does geometry": #182 (historical analogs) and #186 (proposed-and-validated hypotheses) are shipped; the remaining differentiators (#183‚Äì#185, #187‚Äì#190) remain as backlog behind the same invariant.
- **Overlay an alternate count on the chart, side by side (#162).** Elliott work is inherently "primary vs. alternate," but the chart only ever drew one count. Two changes fix that: the auto-analysis now draws the **selected count's own wave line** (its pivots `0‚Üí‚Ä¶‚Üí5` / `A‚ÜíB‚ÜíC`), not just its levels ‚Äî so the ranked count you pick is actually *visible* as a structure ‚Äî and a new **Overlay** control in the auto panel lets you drop a second (alternate) count onto the same chart in a distinct **violet** colour, with its own invalidation/target lines (so each count's levels stay attributable). You see both counts and their differing invalidations at once and judge which the price action favours. Toggling the overlay off cleanly removes its line/markers/levels (no leaked chart primitives), a count can't be its own overlay (the active count is excluded from the overlay picker and dropped if re-selected as primary), and a fresh analysis clears the overlay. Frontend-only ‚Äî the alternates already come down in the response; no new geometry, no LLM. RTL-tested (the overlay picker offers only the non-active counts and toggles the right index; the chart renders the alt line + alt-attributed levels).
- **Place corrective & complex counts by click ‚Äî not impulse-only (#159).** The manual chart could only place an **impulse** (`1‚Äì5`) by click; a correction ‚Äî which is often where an analyst *starts* ‚Äî had to be placed as impulse labels and relabeled afterward via the dropdown. A small **count-type palette** in the chart hint now lets you pick **Impulse (`1‚Äì5`)**, **Zigzag / Flat (`A‚ÄìC`)** or **Complex (`W‚ÄìX‚ÄìY`)** before placing, and each click walks the selected label sequence. The live deterministic verify (REQ-031) then infers and checks the right structure automatically ‚Äî a placed `A‚ÄìB‚ÄìC` reports as **Corrective** and runs the corrective rules ‚Äî because the engine already derives structure from the labels (no backend change; the LLM does no geometry). The type is **locked once the first label is placed** so labels can never mix mid-count; clearing the count re-enables the full palette. Snap-on-place and the ‚óÄ‚ñ∂ nudge are unchanged, and the relabel dropdown stays as a per-label override. Frontend-only, RTL-tested (selecting Zigzag and clicking three points yields `A B C`; the mid-count lock). *(Triangle `A‚ÄìE` placement needs the `D`/`E` labels added to the shared valid-label set front and back ‚Äî tracked as a follow-up.)*
- **Historical analogs ‚Äî "this setup rhymes with N past ones; here's how they resolved" (#182, complete).** The flagship AI differentiator now ships end to end. A **Historical analogs** panel (loaded on demand ‚Äî the sweep is heavy) fingerprints the current count and shows the most similar *past* setups on the same symbol, with their **measured** resolution: hit-rate, the target/invalidated split, median days-to-resolution, and the ranked analogs (each with its date, shape, direction, outcome, time-to-resolve and % match). A new **`GET /api/wave-analysis/analogs`** wires the deterministic engine (feature vector ‚Üí no-lookahead retrieval ‚Üí aggregation) to live candles, builds the corpus from the symbol's own history (cached per day so the cost-heavy sweep runs once), and adds a short **fact-guarded** LLM summary ‚Äî the model may contrast the analogs in prose but cannot cite a rate, count or date the engine didn't compute (`AnalogFactGuard`), and degrades to an explicit reason with no key. This is uniquely defensible: it stands on our measured, leak-free history, which a pure-LLM tool cannot fake. Below a minimum sample the panel says "not enough historical analogs" rather than showing a one-sample rate. Backend service + endpoint are unit- and PostgreSQL-acceptance-tested (incl. the no-lookahead surfacing and the fact-guard); the panel is component-tested; the LLM does no geometry or statistics (ADR-037, REQ-034; extends ADR-009/ADR-027/ADR-028).
- **Historical-analog retrieval ‚Äî the no-lookahead corpus generator (#182).** The second slice of the analog engine: it turns an instrument's candle history into the corpus of past setups the retriever searches. `SetupFeatureExtractor` fingerprints a formed count deterministically ‚Äî direction and confluence strength from the `WaveLevels`, reward:risk and distance-to-invalidation from the same `RiskCalculator` the risk box uses ‚Äî so the query setup and every historical one are described on identical terms. `SetupHistoryBuilder` then replays the deterministic pipeline over history with the **same no-lookahead sweep as the backtest** (`CandleWindow` at each cutoff): each recorded setup carries where it formed, its fingerprint, and how it actually resolved (scored only by the candles that *followed*), so the corpus is structurally incapable of leaking the future. The momentum regimes ride in through a delegate so the sweep stays a pure, testable function. The no-lookahead guarantee is proven by test ‚Äî poisoning the post-cutoff candles cannot change the features of a setup formed before them ‚Äî alongside determinism and the feature-extraction math (both new files 100% covered). - Login now works in local dev: the Vite proxy strips the `Secure` attribute from the session cookie so the browser (http://localhost:5173) stores it. Previously the backend marked the cookie `Secure` over the proxy's HTTPS hop, and the browser dropped it on the insecure dev origin ‚Äî so `/api/auth/me` returned 401 and the app bounced straight back to the login screen
- Restored the frontend test suite after the tooling upgrade by declaring the `@testing-library/dom` peer dependency explicitly (`legacy-peer-deps=true` in `.npmrc` no longer installs it transitively)

### Security
- Google sign-in now rejects logins whose email Google has not verified (`email_verified`). Without this, an attacker controlling a provider account that asserts someone else's address could take over the matching local account (or pre-provision one). Enforced both at the OAuth callback and inside `ExternalLoginAsync` (defense in depth)
- Per-user rate limiting on the expensive endpoints (`/api/wave-analysis`, `/api/market-data`) to curb LLM-cost / upstream abuse, in addition to the login limiter
- Cap the number of annotations per request (prompt-inflation / DoS guard)
- Stop leaking internal details: generic client errors for upstream/LLM failures (including the raw model output), logged server-side instead
- Honour `X-Forwarded-Proto/For` behind a proxy (correct `Secure` cookie + client IP) and enable HSTS outside Development
- Validate the `days` query range on market data (1‚Äì365)
- Stop logging user email addresses; log the non-PII user id instead (CodeQL `cs/exposure-of-sensitive-information`)
- Force the patched `js-yaml` 4.2.0 via npm `overrides` (`npm audit` now reports 0 vulnerabilities)
