## [Unreleased]

### Added
- **Narrative language preference, DE/EN (#228).** Every LLM-authored free-text narrative (auto-analysis rationale, validation notes, analog/sentiment/position summaries, hypothesis reasons, persona commentary) now honours a per-user German-or-English preference. `AppUser` gained a single nullable `NarrativeLanguage` (null = never explicitly chosen, distinct from an explicit English) rather than a new table, since it's one scalar per user. Two seams split by whether the caller already has a `userId`: `INarrativeLanguageProvider` (HttpContext-resolving, mirrors `UserAwareChatClient`'s DI-scope pattern) for the eight narrator/ranker classes with none, `INarrativeLanguageSettingsService` (direct CRUD) for the settings endpoint and `PersonaAnalystPanel`, which already has `userId` in scope. A single pure helper, `NarrativeLanguageDirective.For(language)`, is the one place the German system-prompt instruction is defined; every wired class appends its result the same way `AutoWaveRankRunner`'s persona-guidance suffix already does (#184). Neither literal `*PromptBuilder*.cs` file needed a new parameter — both produce only the user message, so the directive is appended at the (separate, hardcoded) system-message construction site in each Infrastructure-layer caller instead. `EnsembleAutoWaveAnalyzer` resolves the language once per call, reused across every provider in the fan-out. **AC3 needed zero guard changes**: `AnalogFactGuard`/`PositionFactGuard`/`SentimentFactGuard` already match cited figures via a language-agnostic numeric regex — confirmed by six new German-prose regression tests, not by touching any guard. **AC4**: the server never guesses a locale; a new `useNarrativeLanguage` frontend hook suggests a `navigator.language`-derived default the first time it sees a `null` server value and persists it exactly once, after which it behaves like any other explicit preference. New endpoints `GET/PUT /api/settings/narrative-language`. `dotnet test`: 853 passed, 0 failed, 157 skipped (20 new tests). `npx vitest run`: 300 passed, 0 failed (5 new: `useNarrativeLanguage.test.ts`). `npx tsc --noEmit` and `npm run build` clean. New acceptance tests (`NarrativeLanguageAcceptanceTests`) are written to this suite's standard pattern but — like every acceptance test in this session — **could not be executed in this sandbox**, no Docker daemon reachable here (ADR-073, REQ-069).
- **Publication-grade chart export (#227).** The exported PNG (`GET /api/analyses/{id}/chart.png`) lagged the live chart's look — no theme choice, small (1200×700) fixed canvas, no watermark, no disclaimer. `AnnotatedChartComposer`'s colours are now a swappable `ChartPalette` (dark/light), and every fixed-pixel margin/font size scales proportionally to the requested canvas width instead of being hardcoded, so the same draw pipeline serves the new **1920×1080 default** (the publishing-size minimum) and an optional `scale2x` **3840×2160** variant with zero size-specific branching. New optional query params — `theme` (dark/light), `axisScale` (linear/log), `scale2x`, `watermark` (≤64 chars) — each validate to a 400 on a bad value; omitting all of them still renders a valid, now publish-sized PNG, so an existing caller with no query string is unaffected. Added a low-opacity centered watermark (when requested) and a disclaimer line under the title block. Each saved analysis's own persisted alternate scenarios are now drawn as subordinate, lower-opacity zone bands — the exported analogue of the live chart's projection branches, built from data a saved analysis actually has. **Named, deliberate scope gap**: degree-notated nested wave labels and the live chart's specific leg-based projection-branch/retracement-% math cannot be reconstructed for a saved analysis, because a saved analysis never persists the raw pivot sequence either of those are computed from — stated explicitly rather than faked with a look-alike number (ADR-072). `dotnet test`: 833 passed, 0 failed (12 new: composer palette/watermark/alternate-zone/disclaimer/sizing tests, service-level PNG-dimension/pixel/determinism tests; new acceptance tests for the query-param validation are written but — like every acceptance test in this session — could not be executed in this sandbox, no Docker daemon reachable here) (ADR-072, REQ-068).
- **Per-symbol workspace drafts + watchlist (#226).** Switching symbol or timeframe used to unconditionally wipe placed annotations — a careful count on SP500 was gone after a glance at Silver unless formally saved to the track record (meant for finished analyses with outcome tracking, not scratch work). A new server-side `WorkspaceDraft` (one row per user+symbol+interval, annotations/settings stored as opaque JSON the backend never interprets) auto-saves 800ms after the last edit and auto-restores on switch — placing zero annotations deletes rather than persists an empty draft, so a mere glance creates nothing; capped at 50 drafts per user with oldest-first LRU eviction. Drafts are distinct from track-record saves: no outcome tracking, silently overwritten, and deleting one never touches a saved analysis. The hardcoded SP500/NASDAQ/BTC/ETH quick-symbol row is replaced by a user-managed watchlist (`GET/POST /api/watchlist`, `DELETE /api/watchlist/{symbol}`) seeded with those same four symbols on a new user's first read; each entry shows a best-effort last price (reusing the same `IQuoteProvider` seam depot enrichment already uses) and a draft indicator. **Bug found by TDD, fixed before merge**: the restore-completion tracker was originally a plain ref, which doesn't trigger a re-render — for a symbol with no existing draft, nothing else changed state, so the debounced auto-save effect (gated on that same tracker) silently stopped re-arming for the rest of the session; fixed by moving the tracker to React state so restoration settling is itself a reactive dependency. A related fix: the persist handler now uses `async`/`await` instead of chaining `.then()` directly on the save/delete call's return value, which crashed for any caller (including a test double) not returning a real `Promise`. `dotnet test`: 821 passed, 0 failed (new backend acceptance tests for the draft/watchlist round-trip and per-user isolation are written to this suite's standard `AcceptanceWebApplicationFactory` pattern but — like every acceptance test in this session — **could not be executed in this sandbox**, no Docker daemon reachable here; they auto-run once CI or a developer's machine provides Postgres). `npx vitest run`: 295 passed, 0 failed (12 new: `Watchlist.test.tsx` + `WaveWorkspace.test.tsx` draft/watchlist coverage) (ADR-071, REQ-067).
- **Drag pivots directly on the chart (#225, the deferred half of #165).** Placement (click) and fine-tuning (nudge) existed; direct pointer-drag did not, since lightweight-charts v5 has no marker-drag API and a drag's release is, physically, the same gesture as a click. Pivots are hand hit-tested (14px radius) against the `user` wave line's own time+price points — the only series carrying both, converted to pixel coordinates via `chart.timeScale().timeToCoordinate`/`series.priceToCoordinate` — using raw DOM pointer events on the chart container (chart panning disabled for the drag's duration so it never fights the chart's native click-to-pan). On release the dropped position snaps to the nearest candle extreme via the same `snapToCandle` placement and nudge already use, and the debounced live-verify fires once, on drop — a new `displayAnnotations` derivation in `WaveWorkspace` (the committed pivots with the in-flight drag substituted in) feeds the live leg-readout/wave-line rendering without touching the `annotations` state the debounce watches, so a pointermove can never itself trigger a re-verify network call. A drag's release-click is suppressed by a one-shot flag so it never also places a new pivot. Opt-in per `PriceChart` instance — every other consumer is unaffected. **AC1–AC3 unit-tested** (simulated `PointerEvent` gestures asserting the callback contract, incl. that a drag suppresses exactly the next click and a plain click elsewhere is unaffected); **AC4 (hands-on verification in a real browser, including touch) could not be performed in this sandbox** — no live browser was available, the same limitation named throughout this session — stated honestly rather than claimed complete, per the issue's own governance line requiring hands-on verification before merge (ADR-070, REQ-066).
- **Momentum & volume confirmation — RSI/MACD sub-pane + deterministic guidelines (#224).** A professional Elliott Wave read never validates on price geometry alone: wave-5-vs-wave-3 momentum divergence and the wave-3/wave-5 (or wave-B) volume signature are canonical confirmations, and neither reached the count workflow before. `MomentumDivergenceChecker`/`VolumeGuidelineChecker` (pure, static, no DI) add two guideline rows — RSI weaker at the final wave despite the price extension is the confirming divergence; wave 3 (or A/C) outvoluming wave 5 (or B) is the confirming signature — both `Indeterminate`, never a guessed fail, when the data isn't there (missing RSI at the exact pivot date, or every candle in a wave's window reports zero volume, e.g. CoinGecko's free OHLC endpoint). Computed once per candidate set, not inside `WaveGrammarParser`'s beam search (which can evaluate thousands of partitions per parse) — wired into manual Verify and the Auto-analyze/persona-panel candidate generation, the two live paths the issue's own motivation names. A failed guideline measurably reduces `WaveCandidate.Score` via the same `GuidelinePenalty` a parser-internal failed guideline already applies. The two new rule rows reach the LLM prompt automatically (the prompt builders already iterate every rule row under an "AUTHORITATIVE, do NOT re-derive" header) — no new fact-guard plumbing needed. Frontend: an off-by-default RSI sub-pane on the live chart (`lightweight-charts` v5's multi-pane API, the first use of it in this codebase), theme-aware, toggled by a new "RSI" checkbox beside the existing Invalidation/Fib/Targets layer toggles. **Bug found and fixed while wiring this**: reused the same `AutoWaveRankRunner` quota-exception fix from #184's slice, since both tickets touch the same ranking path. `dotnet test`: 821 passed, 0 failed (14 new tests) (ADR-069, REQ-065).
- **Persona panel — the integration slice: prompts, real weighting, endpoint, frontend (#184).** ADR-043 shipped only the deterministic weighting/aggregation core; this closes the named follow-on gap. Three fixed personas (Conservative/Aggressive/Contrarian, `PersonaCatalog`) run sequentially over the *same* `IChatClient` seam every other LLM feature uses (`AutoWaveRankRunner` gained an optional persona-guidance suffix, appended to — never replacing — its base prompt, so a persona can only narrow how the deterministic candidates are read, never introduce one). AC5's cost-bounded degradation is reactive: personas stop the moment one fails on quota exhaustion or any other error, keeping whatever succeeded and reporting `PersonasAttempted` honestly; only the first persona's own failure propagates. `AnalysisSnapshot` gained a nullable `Persona` tag (pure additive migration) set only when saving a persona-panel candidate that exactly one persona independently endorsed as its own top pick (`PersonaRankedCount.EndorsingPersonas`) — an ambiguous save is persisted untagged rather than guessed. `PersonaCalibrationProvider` reuses `TrackRecordService`'s exact candle-fetch-and-evaluate approach, restricted to persona-tagged saves, feeding real measured weights back into future panel runs. New `POST /api/wave-analysis/persona-panel` endpoint and a frontend "Analyst panel" section (consensus score, per-persona weights, endorsing personas) beside the analogs/hypotheses/sentiment sub-panels on the Auto tab.

### Fixed
- **Login rate limit was a single global bucket of 5 req/min, breaking any acceptance-test fixture (or real deployment) with more than a handful of logins in one minute.** Found running #228's acceptance suite in CI: `WatchlistAcceptanceTests` creates ~9 fresh test accounts per fixture run, so logins past the 5th within a 60-second window were silently rejected (429) — and the test helpers (`AcceptanceWebApplicationFactory.AuthenticateAsync`, plus a `NewUserClientAsync` duplicated in three acceptance-test files) never checked the login response's status, so the failure only surfaced several requests later as a confusing `401 Unauthorized` or a `JsonReaderException` parsing an empty body, not as a clear "login failed" error. Raised the `login` policy's limit from 5 to 60 req/min (still a single global bucket by design — partitioning by IP wouldn't have fixed this anyway, since every acceptance-test request shares one loopback address inside `TestServer`) and every affected test helper now calls `EnsureSuccessStatusCode()` on the login response so a future exhaustion fails immediately at its source (ADR-074, REQ-070).
- **Live verify now runs on the chart's own timeframe — weekly/intraday counts no longer get their pivots rejected.** The deterministic re-verify (`POST /api/wave-analysis/verify`) always snapped against ~2y of **daily** candles, regardless of what the chart displayed. A pivot placed on a **weekly** bar carries the week-start date and the weekly extreme — which no single daily candle matches (the extreme prints mid-week, beyond the snapper's ±1-bar tolerance) — so weekly (and 1H/4H) counts were systematically reported as "didn't land on a candle". The request now carries the chart's **interval**, and `WaveVerificationService` fetches/resamples the **same series the chart showed** (daily→weekly via the existing `CandleResampler`; intraday via the intraday provider, with a source-appropriate lookback). Unit-tested both ways: weekly pivots snap on the weekly series, and the old daily-only behaviour is pinned as the bug this guards against.
- **Auto-analysis no longer fails on a truncated/malformed LLM ranking.** Gemini 2.5 spends "thinking" tokens against the output budget and, with several candidates and verbose rationales, cut its ranking JSON off mid-string → the whole analysis failed with a 502 and no retry. The ranking runner now detects an empty/truncated/malformed response and retries **once with corrective feedback** (asking for the complete JSON and brevity — same discipline as the vision extractor), with a doubled token budget; the prompt itself now demands concise rationales so truncation is rare in the first place. Token usage is summed across attempts. Unit-tested (truncated-then-valid recovers; malformed-twice throws after exactly one retry).
- **A canceled request no longer logs as an error 500.** When the client aborts a fetch (e.g. React StrictMode double-mount, navigating away), the escaping `OperationCanceledException` was logged as an error and returned as 500. The global exception handler now recognizes the client-abort case and responds 499 ("client closed request") at debug log level — real faults still log and return 500.

- **Top-down multi-timeframe consistency (REQ-023).** A new deterministic read, `GET /api/wave-analysis/topdown`, counts a symbol across a weekly → daily → 4-hour ladder and makes each finer count *live inside* the wave unfolding on the timeframe above it: counts travelling the wrong direction are rejected, and a class or price-window mismatch is penalized. Each link carries a verdict — **Consistent**, **Tension** or **Contradiction** — with a reason, so the daily and weekly reads of the same instrument can no longer silently disagree. All pure/static logic (`WaveContextDeriver`, `WaveContextConstraint`, `TopDownWaveAnalyzer`), unit-tested including determinism; a timeframe an instrument can't serve (e.g. no intraday source for 4H) is skipped honestly. The auto-analysis panel shows a compact `1W → 1D → 4H` breadcrumb with a verdict badge per link, above the counts. No LLM and no token cost — the LLM still does no geometry (ADR-009/ADR-024).
- **Log-correct Fibonacci math + scored confluence zones (REQ-022).** Fibonacci retracements and extensions are now computed in **log** price space when a count spans more than ~3× its low (auto-selected and always reported), so the levels match what a professional draws on a log chart instead of the arithmetic midpoint. Every projection also carries **scored confluence zones** — the "green boxes" where several Fibonacci ratios (ideally across wave degrees) stack up — ranked by the summed degree weight of their contributing levels, each labelled with its ratio, leg and scale. Wave 5 targets draw confluence from two legs (Wave 1 and net Waves 1–3). All new logic is pure/static (`FibMath`, `FibConfluenceCalculator`) and unit-tested against hand-computed values; `WaveLevels` gains `scale` + `confluenceZones` in the API and frontend contract, and `LevelsSummary` renders the scale badge and each zone with its score and contributing levels. The LLM still does no geometry (ADR-009/ADR-023) — it only narrates the deterministic zones.
- **Per-user encrypted API-key vault (REQ-012).** LLM API keys are now stored **server-side, encrypted at rest** with ASP.NET Core Data Protection and never returned to the client — replacing the previous `localStorage`-only facade, so the Settings page's security promise is finally true. New `/api/keys` endpoints (list / save / delete / set-default) with per-user isolation; the frontend Settings page reads and writes the vault. (Consuming the stored key in the LLM pipeline is tracked as a follow-up.)
- **Timeframe selector — Daily / Weekly (REQ-010).** A Daily/Weekly toggle on the chart; `GET /api/market-data/{symbol}?interval=1d|1w` returns the selected timeframe. Weekly bars are aggregated deterministically from the daily candles (pure, unit-tested `CandleResampler`), and RSI/MACD are computed on the selected timeframe. (4H needs an intraday data source — tracked as a follow-up.)
- **Confidence calibration (REQ-008).** `GET /api/analyses/calibration` reports, per AI confidence level (high/medium/low), how many of your saved analyses concluded and how many reached their target — with a hit rate per level and overall — so "high confidence" is backed by your own track record. Pure, unit-tested aggregation over the live-evaluated outcomes; documented in the Scalar/OpenAPI UI
- **Price alerts (REQ-007).** An opt-in scheduled background pass (`Alerts:Enabled`, cron) re-evaluates your still-pending saved analyses and, when one is invalidated or reaches its target, delivers a one-time alert (chart + caption) through the same channels as the daily report (Telegram/Email). Each transition fires exactly once — the snapshot records the outcome it last alerted on. The alert decision is a pure, unit-tested function; delivery is to the operator-configured channels (per-user delivery targets are a follow-up)
- **Track-record history UI (REQ-006).** Each ranked count in the auto-analysis panel now has a **Save** action, and a new **Track record** panel lists your saved analyses (newest first) with an outcome badge — Pending / Invalidated / Target reached — plus the invalidation, target and last-evaluated price, and a delete action. Consumes the existing `/api/analyses` endpoints; wired via TanStack Query with cache invalidation on save/delete
- **Track record (persistence).** Save an Elliott Wave analysis to a personal, per-user track record and later see whether its invalidation held or its target was reached. New endpoints: `POST /api/analyses` (save the count's direction, invalidation line and target zone), `GET /api/analyses` (list newest-first, each with an outcome — `Pending` / `Invalidated` / `TargetReached` — evaluated fresh against the candles since it was saved), `DELETE /api/analyses/{id}` (owner-scoped). Backed by a new `AnalysisSnapshot` table on PostgreSQL (EF migration) and a pure, wick-aware `AnalysisOutcomeEvaluator` (first-event-wins, invalidation breaks the tie). First step toward AI-confidence calibration and price alerts
- **Wave grammar parser (nested, multi-degree counts).** Elliott counting is now parsing: the rulebook is modelled as a grammar (Motive → Impulse|Diagonal; Corrective → Zigzag|Flat|Triangle; each wave a terminal leg or a nested structure) and parsed via memoized dynamic programming over pivot intervals with beam search. Hard rules prune, guideline scoring (Fibonacci fit, wave-2/4 alternation, channel linearity, time proportion — tunable via `WaveScoringOptions`) ranks. The full-auto analysis now returns nested counts with per-node structure, degree and score (`tree` + `score` on each ranking, additive), plus a `searchTruncated` flag when the evaluation budget bounded the search
- **Corrective and diagonal rule checkers.** Deterministic checkers for zigzag (5-3-5), flat (3-3-5 with regular/expanded/running classification), contracting triangle and contracting diagonal (wave-4 overlap allowed — fixes valid diagonals failing the impulse rules). `RuleResult` gains an additive `isGuideline` flag: failed guidelines flavor a count, only hard rules invalidate
- **Corrective projections.** Forward levels for unfolding B/C waves of zigzags and flats, triangle legs (contraction barrier as invalidation) and the post-triangle thrust; complete corrections project the recovery zone and invalidate beyond wave C
- **Wick-aware, adaptive, multi-scale pivot detection.** The ZigZag detector confirms reversals against candle highs/lows (pivots land on true wick extremes), optionally scales its threshold with volatility (`DetectAtrAdaptive`, k×ATR) and can detect several Elliott degrees at once (`DetectMultiScale`, coarse scales guaranteed subsets of finer ones)
- LLM calls request native JSON mode (`ChatResponseFormat.Json`) where the provider supports it; the robust extraction fallback remains for providers that ignore the hint
- **Google sign-in (end-to-end).** "Continue with Google" on the login screen; the OAuth callback provisions the account just-in-time and issues the same opaque session cookie as password login. Shown only when Google OAuth is configured (`Authentication:Google:ClientId`), surfaced to the frontend via `GET /api/auth/providers`. Post-login redirect is configurable (`Authentication:Google:PostLoginRedirectUri`)
- Deterministic `ElliottRuleChecker`: the three hard Elliott rules + Fibonacci ratios computed in code (objective, no LLM) and returned alongside the AI assessment as `ruleReport`

- Google sign-in now rejects logins whose email Google has not verified (`email_verified`).
