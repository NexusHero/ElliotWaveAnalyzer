## [Unreleased]

### Added
- **Alternate hypotheses — the AI proposes structures, the engine validates them (#186; concludes the AI-differentiators epic #191).** The purest expression of the moat: the LLM **proposes which Elliott structures are worth testing** (impulse, diagonal, zigzag, flat, triangle — a bounded vocabulary), and the **deterministic engine generates and rule-checks each** with the same positional checkers the beam parser uses. Validated hypotheses are shown with their guideline score; **rejected ones are shown as "considered and rejected" with the exact hard rule they violated** — never as valid counts. The model never asserts a count it didn't earn against the rules: an out-of-vocabulary suggestion is dropped **before** generation, the number tested is **capped** (no unbounded prompting), and with no LLM key the feature is simply absent (the deterministic search is untouched). A new **`GET /api/wave-analysis/hypotheses`** and an **Alternate hypotheses** panel (on-demand) drive it. Every rule verdict is deterministic and reproducible; the LLM proposes, the engine owns validity and scoring (ADR-038, REQ-035, §6 Scenario 22). Backend engine (unit + PostgreSQL acceptance tested, 91.98% coverage) + frontend panel. **This is the capstone of epic #191** — the AI-differentiator moat, all built on "the LLM never does geometry": #182 (historical analogs) and #186 (proposed-and-validated hypotheses) are shipped; the remaining differentiators (#183–#185, #187–#190) remain as backlog behind the same invariant.
- **Overlay an alternate count on the chart, side by side (#162).** Elliott work is inherently "primary vs. alternate," but the chart only ever drew one count. Two changes fix that: the auto-analysis now draws the **selected count's own wave line** (its pivots `0→…→5` / `A→B→C`), not just its levels — so the ranked count you pick is actually *visible* as a structure — and a new **Overlay** control in the auto panel lets you drop a second (alternate) count onto the same chart in a distinct **violet** colour, with its own invalidation/target lines (so each count's levels stay attributable). You see both counts and their differing invalidations at once and judge which the price action favours. Toggling the overlay off cleanly removes its line/markers/levels (no leaked chart primitives), a count can't be its own overlay (the active count is excluded from the overlay picker and dropped if re-selected as primary), and a fresh analysis clears the overlay. Frontend-only — the alternates already come down in the response; no new geometry, no LLM. RTL-tested (the overlay picker offers only the non-active counts and toggles the right index; the chart renders the alt line + alt-attributed levels).
- **Place corrective & complex counts by click — not impulse-only (#159).** The manual chart could only place an **impulse** (`1–5`) by click; a correction — which is often where an analyst *starts* — had to be placed as impulse labels and relabeled afterward via the dropdown. A small **count-type palette** in the chart hint now lets you pick **Impulse (`1–5`)**, **Zigzag / Flat (`A–C`)** or **Complex (`W–X–Y`)** before placing, and each click walks the selected label sequence. The live deterministic verify (REQ-031) then infers and checks the right structure automatically — a placed `A–B–C` reports as **Corrective** and runs the corrective rules — because the engine already derives structure from the labels (no backend change; the LLM does no geometry). The type is **locked once the first label is placed** so labels can never mix mid-count; clearing the count re-enables the full palette. Snap-on-place and the ◀▶ nudge are unchanged, and the relabel dropdown stays as a per-label override. Frontend-only, RTL-tested (selecting Zigzag and clicking three points yields `A B C`; the mid-count lock). *(Triangle `A–E` placement needs the `D`/`E` labels added to the shared valid-label set front and back — tracked as a follow-up.)*
- **Historical analogs — "this setup rhymes with N past ones; here's how they resolved" (#182, complete).** The flagship AI differentiator now ships end to end. A **Historical analogs** panel (loaded on demand — the sweep is heavy) fingerprints the current count and shows the most similar *past* setups on the same symbol, with their **measured** resolution: hit-rate, the target/invalidated split, median days-to-resolution, and the ranked analogs (each with its date, shape, direction, outcome, time-to-resolve and % match). A new **`GET /api/wave-analysis/analogs`** wires the deterministic engine (feature vector → no-lookahead retrieval → aggregation) to live candles, builds the corpus from the symbol's own history (cached per day so the cost-heavy sweep runs once), and adds a short **fact-guarded** LLM summary — the model may contrast the analogs in prose but cannot cite a rate, count or date the engine didn't compute (`AnalogFactGuard`), and degrades to an explicit reason with no key. This is uniquely defensible: it stands on our measured, leak-free history, which a pure-LLM tool cannot fake. Below a minimum sample the panel says "not enough historical analogs" rather than showing a one-sample rate. Backend service + endpoint are unit- and PostgreSQL-acceptance-tested (incl. the no-lookahead surfacing and the fact-guard); the panel is component-tested; the LLM does no geometry or statistics (ADR-037, REQ-034; extends ADR-009/ADR-027/ADR-028).
- **Historical-analog retrieval — the no-lookahead corpus generator (#182).** The second slice of the analog engine: it turns an instrument's candle history into the corpus of past setups the retriever searches. `SetupFeatureExtractor` fingerprints a formed count deterministically — direction and confluence strength from the `WaveLevels`, reward:risk and distance-to-invalidation from the same `RiskCalculator` the risk box uses — so the query setup and every historical one are described on identical terms. `SetupHistoryBuilder` then replays the deterministic pipeline over history with the **same no-lookahead sweep as the backtest** (`CandleWindow` at each cutoff): each recorded setup carries where it formed, its fingerprint, and how it actually resolved (scored only by the candles that *followed*), so the corpus is structurally incapable of leaking the future. The momentum regimes ride in through a delegate so the sweep stays a pure, testable function. The no-lookahead guarantee is proven by test — poisoning the post-cutoff candles cannot change the features of a setup formed before them — alongside determinism and the feature-extraction math (both new files 100% covered). The endpoint, the fact-guarded LLM narrator and the frontend panel are the remaining integration. Backend core + tests only — no runtime endpoint yet (ADR-037; extends ADR-027).
- **Historical-analog retrieval — the deterministic core (#182).** Groundwork for the flagship "this setup rhymes with N past ones; here's how they resolved" panel — and it's defensible precisely because it stands on our *measured, no-lookahead* history, which a pure-LLM tool cannot fake. This first slice ships the **pure, LLM-free engine**: a deterministic **feature vector** that fingerprints a setup (structure, direction, guideline score, confluence strength, reward:risk, distance-to-invalidation, RSI/MACD regime; `SetupFeatureVector`, cosine similarity, no `pgvector` dependency — in-memory kNN); a **retriever** (`AnalogRetriever`) that returns the k nearest *past* setups, restricted to ones that **concluded strictly before** the query's as-of date (**no-lookahead**, the same guarantee as the backtest) and **only concluded** (a still-open setup is not evidence); an **aggregator** (`AnalogAggregator`) that computes hit-rate, the target/invalidated split and the median resolution time **only from concluded analogs**, flagging "insufficient history" below a minimum so a one-sample rate is never shown as reliable; and an **analog fact-guard** (`AnalogFactGuard`, a sibling of the portfolio narrator's guard) that rejects any summary citing a rate, count or date the engine didn't compute. Ranking, no-lookahead, determinism, insufficient-history and the fact-guard are exhaustively unit-tested (the new logic is 100% covered). The corpus wiring (from the no-lookahead backtest sweep), the `GET /api/wave-analysis/analogs` endpoint, the fact-guarded LLM narrator and the frontend panel are the follow-on integration behind these seams. Backend core + tests only — no runtime endpoint yet (ADR-037; extends ADR-009/ADR-027/ADR-028).
- **The "green boxes" now shade the live chart — confluence/entry/target zones as bands (#158).** The engine's scored confluence zones and the entry/target zones used to exist only as two bare axis lines plus a text list (and in the exported PNG); the live chart now fills them as **semi-transparent shaded bands** between each zone's low and high, so the analyst *sees* where price is expected to react instead of reading numbers. Entry/pullback bands are blue, target bands green — the **same fills as the annotated-PNG export** (`AnnotatedChartComposer`), so the live chart and the exported image agree on the same geometry. Bands track the axis through the **log/linear switch** (they map price→pixel through the series, so they line up with the log-correct Fibonacci math), redraw on theme change, and are cleaned up without leaking on every update. The band geometry is a pure, unit-tested helper (`levelsToZoneBands`, honouring the existing layer toggles and tying each confluence zone to the toggle matching its kind); the drawing lives in a confined Lightweight-Charts series primitive (`ZoneBandsPrimitive`) so the chart stays a thin renderer over deterministic data — the LLM still does no geometry. Frontend-only, no backend change (ADR-036). Completes #158 (the log/linear toggle shipped earlier).
- **Mutation testing in CI — proof the tests actually catch bugs (#195).** Coverage proves the deterministic core is *run*; it can't prove a test would *fail* if the behaviour changed. **Stryker** now rewrites the pure algorithmic core (`Application/**` — the geometry, Elliott rules, Fibonacci math, risk, scanner, projections, scoring — plus `Domain/CandleWindow`, the no-lookahead guard, but **excluding the request-orchestration services** that are acceptance-tested rather than unit-tested) with small faults (flip a comparison, drop a statement, swap `&&`/`||`) and re-runs the suite: a mutant that **survives** is a behaviour no test pins down. It runs **nightly** (`.github/workflows/mutation.yml`, plus on demand) against the same Postgres the backend job uses, uploads the HTML/JSON report as an artifact and appends the Markdown survivor list (file · line · mutation) to the run summary, and **fails below an enforced score floor** (`thresholds.break`) so the suite's strength on the core is a measured, ratcheted number rather than a coverage guess. Fixes the previously-dormant Stryker config, which used pre-4.x key names (`threshold-*`, `excluded-mutations`) that Stryker 4.16 rejects — it never actually ran; it now shares one gate between a local `dotnet stryker` and CI. Scoped and string/LINQ-filtered so the run stays bounded. Test-infrastructure and docs only — no runtime change (ADR-035, extends ADR-034/ADR-015).
- **Logarithmic price axis toggle — the chart matches the log-correct math (part of #158).** A **Log** toggle in the chart header switches the price axis between linear and logarithmic. It **auto-enables** when the active count was computed in log space (a wide count auto-selects log Fibonacci scaling), so the drawn levels finally line up with where the math puts them — and the analyst can override it either way. Higher-degree counts spanning several multiples of price now read correctly instead of looking distorted on a linear axis. (The shaded confluence-zone boxes, the other half of #158, now ship too — see above.)
- **Save & export your own edited count (#157).** Closes the gap from the analyst-in-the-loop editor (#151): the live deterministic verdict now carries a **Save to track record** action, so a hand-drawn count you edit is persisted like any other analysis — with its deterministic invalidation and target zones — and appears in the Track record panel with an outcome that evaluates over time (recorded with a `manual` confidence, since a hand-drawn count has no LLM ranking). Once saved, a **Download chart** link exports the annotated PNG of your count (the existing per-analysis renderer). The button re-enables automatically on any further edit, so an edited count is never silently the same as the saved one. The geometry mapping is a shared, unit-tested helper (`verificationToTrackRequest`) reusing the same levels→scenario logic as the ranked-count save. Frontend-only, reusing the existing persistence and export endpoints.
- **See the count as a structure — connected wave lines on the chart (#156).** The chart now draws a **connected polyline through your pivots** (0→1→2→3→4→5, A→B→C, …) instead of showing labeled dots only, so the wave *structure* — the impulse zig-zag, the retracement depth, wave 3 dwarfing 1 and 5 — is finally visible at a glance. Your count draws in the user colour and the AI's own count draws its own line in amber, so the two reads can be compared side by side; both follow the active light/dark theme and redraw live as pivots are added, nudged, relabeled or deleted. The point derivation is a pure, unit-tested helper (`toWaveLinePoints`) and the chart stays a thin renderer over it; labels remain on top and legible. Frontend-only, no backend change.
- **No-lookahead & LLM-swap invariants, property-based (REQ-032, extends the test foundation).** Two headline safety guarantees are now attacked by generated variants: **(1) no-lookahead** — for any series, any cutoff and any *poisoned* future appended beyond it, the deterministic analysis over the `CandleWindow` is byte-identical to the analysis over only the visible candles (the future physically cannot leak); and **(2) LLM-swap invariance (I1, "the LLM never does geometry")** — an end-to-end acceptance test swaps the model's *entire* output and asserts every deterministic field (the rule report and the forward levels) stays byte-identical while only the LLM's coaching text changes. If a model could ever move the geometry, this test fails. No runtime change.
- **Property-based test foundation — invariants over examples (REQ-032).** The safety invariants that make the tool defensible are now encoded as **property-based + metamorphic** tests (via **CsCheck**) that attack the pure deterministic core (`RiskCalculator`, `WaveVerifier`, `PivotSnapper`, `ElliottRuleChecker`) with **thousands of generated, automatically-shrunk** variants — not a handful of hand-picked cases. Generators emit only *valid* fixtures (OHLC with `Low ≤ Open,Close ≤ High`; annotations on real candle extremes), so a failure is always a real defect. The first batch proves: position sizing is **never negative** (and never a precision underflow on realistic inputs), no-valid-stop **⇔** the direction guard, the verifier **never throws** with every snapped pivot on a **real candle** and `IsValid` **⇔** no hard rule failed, snapping is **idempotent**, the read is **deterministic**, and the rule verdict is **invariant under price-scaling and time-shift** (a count is about structure, not absolute price/dates). This is the foundation of the broader test strategy (epic #201): adversarial prompt-injection/fact-guard corpora, mutation testing, browser-E2E, frontend↔backend contract-drift checks, real-LLM nightly eval and load/soak are tracked as follow-on issues (ADR-034). Test-and-docs only — no runtime change.
- **Analyst-in-the-loop editing — the analyst works, the tool verifies (REQ-031).** The manual chart is now a workbench: **place, nudge (◀ ▶), relabel and delete** pivots, and on every (debounced) edit the **deterministic** pipeline re-runs and a live **Live check** panel shows the objective read — valid / rule-violation, the failing hard rules by name, the guideline score, any pivots that didn't land on a candle, and the forward projections (invalidation, support/target zones, confluence, channels). A new **`POST /api/wave-analysis/verify`** does this with **no LLM and no token usage** (the LLM is only offered afterwards to narrate the analyst's *own* count — it never edits the geometry). Every edit **snaps to a real candle extreme** — client-side for instant feedback and again authoritatively on the server (`PivotSnapper`), so a moved pivot always lands on real data and pivots that don't snap are reported, not trusted. The edited count reuses the existing track-record persistence and annotated-chart export. Pure verifier (snap → hard rules → projections → score) is unit-tested (impulse, corrective, rule-violation, off-candle, determinism); the endpoint is a PostgreSQL acceptance test; the snap helpers and the panel are covered by frontend tests (ADR-009/ADR-033).
- **Risk layer — from a count to a sized trade (REQ-030).** A new **`POST /api/risk`** turns a count's geometry into risk terms: given an entry, the count's invalidation (the natural stop), the target prices and the direction, plus an account-risk input (**percent of equity or an absolute amount**), it computes **stop distance** (absolute + %), **reward:risk against each target**, and the **position size that risks exactly the chosen capital** — with the notional deployed. Direction-aware (long vs short) and fully guarded: an entry on the wrong side of the invalidation returns an explicit `hasValidStop: false` with a reason (never a negative or infinite size), and a zero/negative account-risk keeps the stop and R:R but omits the size (no divide-by-zero). A compact **risk box** on the levels panel collects the entry (defaulting to current price), account and risk %, and shows the stop, size and R:R per target — mapping each target *zone* to its first-touch edge. **No LLM** — pure arithmetic on your own inputs, with an explicit "not trading advice" disclaimer. The calculator is unit-tested against hand-computed fixtures (incl. the wrong-side and zero-risk guards); the endpoint is a PostgreSQL acceptance test and the risk box a component test (ADR-009/ADR-032).
- **Your own LLM key now drives your analyses (REQ-013).** The encrypted per-user key vault (REQ-012) is finally *consumed*: the active-provider chat client is resolved **per request** against the calling user's decrypted key for that provider, so a user who saved their own Gemini/Claude/OpenAI key gets their own quota and billing across every LLM-using feature (manual validation, single-provider ranking, the portfolio narrator, vision import) — transparently, with no call-site changes. A user without a stored key, or a background job with no signed-in user, falls back to the operator's startup key exactly as before. All provider-SDK construction is consolidated into one `IUserChatClientFactory` shared by the startup and per-user paths (DRY); the key is decrypted only to build the client and is **never logged**. The routing and the provider-construction seam are unit-tested; the save-key-then-analyze path is a PostgreSQL acceptance test (ADR-031). *Known limitation:* the multi-provider **Ensemble** ranker still uses the operator keys.
- **Setup scanner — sweep the universe for setups in one click (REQ-029).** A new **`GET /api/scan`** runs the deterministic count pipeline across a set of symbols (comma-separated `symbols`, or a configured default universe) and returns the matching setups **ranked most-relevant first** — price already inside a zone, then higher guideline score, then tighter risk (closer to invalidation). Each hit carries the structure, unfolding wave, direction, score, current price, distance-to-invalidation and in-zone flags; filters (`structure`, `minScore`, `inZone`) narrow the set. **No LLM** — cheap enough to sweep many symbols — with bounded concurrency, a hard symbol cap and a per-(symbol, day) cache; a symbol the data source can't serve is skipped, never aborting the scan, and the response reports `scanned`/`matched` so coverage is explicit (no silent truncation). A new **Scanner** panel drives it. Pure scanner + ranking + filter logic is unit-tested; the end-to-end sweep is a PostgreSQL acceptance test (ADR-009/ADR-030).
- **Vision import — verify any analyst's chart against the rules and the real data (REQ-028).** Upload an annotated chart screenshot to **`POST /api/wave-analysis/verify-image`** (multipart, size/type-limited like depot import; the image is parsed in-request and **never stored**) and, optionally, confirm the instrument/timeframe. A vision LLM extracts the *claimed* count in strict JSON (schema-validated, one retry), then the deterministic engine takes over: **every claimed pivot must snap to a real candle extreme** (±0.5% price / ±1 bar) or it is rejected with a reason (`"claimed pivot at 64.86 on Jul 3 — no such extreme within ±0.5%"`), never trusted; if too few survive, the report says the image *could not be reliably extracted* rather than guessing. The hard Elliott rules then verify what survived (violations reported by name), side-by-side with our own best count on the same window (structures, our score, our confluence zones, agree/differ). A **Verify a chart** panel offers the upload + report view. LLM for perception, rules for judgment — swapping the vision model can't change a single verdict (ADR-009/ADR-029). All snap/guard/rule/comparison logic is pure and unit-tested; the end-to-end path is a PostgreSQL acceptance test with a faked vision client.
- **Portfolio auto-commentary — a professional review of your whole depot, automatically (REQ-027).** A new **`GET /api/depot/analysis`** reviews every holding in your imported depot: it resolves the ISIN to an instrument, runs the deterministic top-down count chain, derives the scenario geometry (invalidation, entry and target zones), classifies where current price sits (above/below invalidation, in the entry zone), and adds a short LLM narrative **written strictly from those facts**. The narrative passes through a pure `PositionFactGuard` that rejects any text citing a price not in the fact sheet — so the model can't invent a target — and degrades gracefully to an explicit reason when no LLM key is configured or the fact-guard fires; the deterministic brief always stands. A portfolio **summary** aggregates the risk read across positions, and holdings that can't be resolved or analyzed are returned in an explicit **unresolved** list with a reason — never a silent gap. Results are cached per (position, day) so re-opening is cheap, and an **opt-in** scheduled refresh (`PortfolioReview:Enabled`) warms every depot on a cron. A new **Portfolio review** panel shows the cards + summary. Pure orchestration/guard/summary logic is unit-tested (including the anti-hallucination guard and no-key degradation); the end-to-end path is a PostgreSQL acceptance test. The LLM only narrates validated facts (ADR-009/ADR-028).
- **Backtest harness with a no-lookahead guarantee — the credibility engine (REQ-026).** The whole deterministic pipeline (pivots → parse → scenario tree with zones/invalidations) is now replayed over **history**: a cutoff slides forward and, at each step, the analysis sees **only** the candles up to the cutoff — enforced structurally by a `CandleWindow` guard type whose indexer throws for any index at or beyond the cutoff — while the following candles score the recorded scenario with the existing outcome semantics. Measured hit rates aggregate by structure, confidence, confluence and timeframe (open scenarios excluded from the denominator), are persisted idempotently (keyed by a dataset hash, so a re-run never duplicates), and read back via **`GET /api/backtest/summary`**; the track-record page shows a "Measured performance" panel. The harness's per-confidence rates also feed **priors into scenario probabilities** (REQ-024): a thin personal record falls back to the backtest prior (`ProbabilityBasis.Backtested`), a rich one blends toward it but stays led by the user's own measured rate. Running a backtest is a Development-only `POST /api/backtest/run` (404 in production). **The no-lookahead property is enforced by tests** — a poisoned future must not change any earlier recorded scenario, and a fully-scored result must be identical on the truncated series. Parameter optimization is deliberately out of scope (overfitting). New EF migration `AddBacktestRuns`. Engine and aggregation are pure and unit-tested; the end-to-end run + idempotency is a PostgreSQL acceptance test. No LLM (ADR-009/ADR-027).
- **Channel projections + publication-grade annotated chart export (REQ-025).** Every projection now carries deterministic **Elliott channels**: the base channel (0→2 line, parallel through wave 1) once wave 2 exists and the acceleration channel (2→4 line, parallel through wave 3) once wave 4 exists — the latter projecting a **wave-5 target band** one acceleration-leg beyond wave 4. Lines are fit in log space on a log-scaled count (a straight channel on a log chart stays straight), asserted against hand-computed slope/intercept. Any saved analysis can be exported as an annotated PNG via **`GET /api/analyses/{id}/chart.png`** (auth, per-user, 404 for another user's id): candles, shaded entry/target zone boxes with edge prices, the invalidation line with a price tag, scenario arrows (primary solid, alternates dashed) and a title block (symbol · timeframe · date · scale). Rendering goes through a backend-agnostic **draw-op seam** — a pure Application `AnnotatedChartComposer` decides all geometry and emits an ordered `ChartScene`, and a confined SkiaSharp backend only rasterizes it — so the layout is unit-tested structurally (no OCR) and the output is **deterministic** (byte-identical for the same analysis + render date). The track-record UI gains a per-row chart download. SkiaSharp stays entirely in Infrastructure; the LLM still does no geometry (ADR-009/ADR-026).
- **Scenario tree with calibrated probabilities, zone-entry alerts and invalidation auto-switch (REQ-024).** A saved analysis is now a **tree** — a primary count plus up to two alternates — each with its entry/target zones, hard invalidation, and a probability drawn from *your own* measured track record (or an explicit "insufficient data" marker when fewer than 10 concluded analyses back it, never an invented number). When price **enters the entry zone** you get a one-time "entry zone reached" alert. When price **breaks the primary's invalidation**, the system auto-switches: it promotes the best-scored alternate to primary, retires the old primary (kept for the audit trail), records an append-only switch event, and re-opens the analysis under the new primary — or concludes it invalidated if no alternate remains. The `GET /api/analyses` shape carries the tree (per-scenario zones + probability/insufficient-data marker) and the switch history; the track-record UI renders both. New EF migration `AddScenarioTree`. Decision logic (`ScenarioProbability`, `ScenarioSwitch`, `ZoneEntryDecision`) is pure and unit-tested; the full save→invalidate→switch lifecycle is covered by a PostgreSQL acceptance test. No LLM chooses a count (ADR-009/ADR-025).
- **Top-down multi-timeframe consistency (REQ-023).** A new deterministic read, `GET /api/wave-analysis/topdown`, counts a symbol across a weekly → daily → 4-hour ladder and makes each finer count *live inside* the wave unfolding on the timeframe above it: counts travelling the wrong direction are rejected, and a class or price-window mismatch is penalized. Each link carries a verdict — **Consistent**, **Tension** or **Contradiction** — with a reason, so the daily and weekly reads of the same instrument can no longer silently disagree. All pure/static logic (`WaveContextDeriver`, `WaveContextConstraint`, `TopDownWaveAnalyzer`), unit-tested including determinism; a timeframe an instrument can't serve (e.g. no intraday source for 4H) is skipped honestly. The auto-analysis panel shows a compact `1W → 1D → 4H` breadcrumb with a verdict badge per link, above the counts. No LLM and no token cost — the LLM still does no geometry (ADR-009/ADR-024).
- **Log-correct Fibonacci math + scored confluence zones (REQ-022).** Fibonacci retracements and extensions are now computed in **log** price space when a count spans more than ~3× its low (auto-selected and always reported), so the levels match what a professional draws on a log chart instead of the arithmetic midpoint. Every projection also carries **scored confluence zones** — the "green boxes" where several Fibonacci ratios (ideally across wave degrees) stack up — ranked by the summed degree weight of their contributing levels, each labelled with its ratio, leg and scale. Wave 5 targets draw confluence from two legs (Wave 1 and net Waves 1–3). All new logic is pure/static (`FibMath`, `FibConfluenceCalculator`) and unit-tested against hand-computed values; `WaveLevels` gains `scale` + `confluenceZones` in the API and frontend contract, and `LevelsSummary` renders the scale badge and each zone with its score and contributing levels. The LLM still does no geometry (ADR-009/ADR-023) — it only narrates the deterministic zones.
- **Per-user encrypted API-key vault (REQ-012).** LLM API keys are now stored **server-side, encrypted at rest** with ASP.NET Core Data Protection and never returned to the client — replacing the previous `localStorage`-only facade, so the Settings page's security promise is finally true. New `/api/keys` endpoints (list / save / delete / set-default) with per-user isolation; the frontend Settings page reads and writes the vault. (Consuming the stored key in the LLM pipeline is tracked as a follow-up.)
- **Timeframe selector — Daily / Weekly (REQ-010).** A Daily/Weekly toggle on the chart; `GET /api/market-data/{symbol}?interval=1d|1w` returns the selected timeframe. Weekly bars are aggregated deterministically from the daily candles (pure, unit-tested `CandleResampler`), and RSI/MACD are computed on the selected timeframe. (4H needs an intraday data source — tracked as a follow-up.)
- **Confidence calibration (REQ-008).** `GET /api/analyses/calibration` reports, per AI confidence level (high/medium/low), how many of your saved analyses concluded and how many reached their target — with a hit rate per level and overall — so "high confidence" is backed by your own track record. Pure, unit-tested aggregation over the live-evaluated outcomes; documented in the Scalar/OpenAPI UI
- **Price alerts (REQ-007).** An opt-in scheduled background pass (`Alerts:Enabled`, cron) re-evaluates your still-pending saved analyses and, when one is invalidated or reaches its target, delivers a one-time alert (chart + caption) through the same channels as the daily report (Telegram/Email). Each transition fires exactly once — the snapshot records the outcome it last alerted on. The alert decision is a pure, unit-tested function; delivery is to the operator-configured channels (per-user delivery targets are a follow-up)
- **Track-record history UI (REQ-006).** Each ranked count in the auto-analysis panel now has a **Save** action, and a new **Track record** panel lists your saved analyses (newest first) with an outcome badge — Pending / Invalidated / Target reached — plus the invalidation, target and last-evaluated price, and a delete action. Consumes the existing `/api/analyses` endpoints; wired via TanStack Query with cache invalidation on save/delete
- **Track record (persistence).** Save an Elliott Wave analysis to a personal, per-user track record and later see whether its invalidation held or its target was reached. New endpoints: `POST /api/analyses` (save the count's direction, invalidation line and target zone), `GET /api/analyses` (list newest-first, each with an outcome — `Pending` / `Invalidated` / `TargetReached` — evaluated fresh against the candles since it was saved), `DELETE /api/analyses/{id}` (owner-scoped). Backed by a new `AnalysisSnapshot` table on PostgreSQL (EF migration) and a pure, wick-aware `AnalysisOutcomeEvaluator` (first-event-wins, invalidation breaks the tie). First step toward AI-confidence calibration and price alerts
- **Wave grammar parser (nested, multi-degree counts).** Elliott counting is now parsing: the rulebook is modelled as a grammar (Motive → Impulse|Diagonal; Corrective → Zigzag|Flat|Triangle; each wave a terminal leg or a nested structure) and parsed via memoized dynamic programming over pivot intervals with beam search. Hard rules prune, guideline scoring (Fibonacci fit, wave-2/4 alternation, channel linearity, time proportion — tunable via `WaveScoringOptions`) ranks. The full-auto analysis now returns nested counts with per-node structure, degree and score (`tree` + `score` on each ranking, additive), plus a `searchTruncated` flag when the evaluation budget bounded the search
- **Corrective and diagonal rule checkers.** Deterministic checkers for zigzag (5-3-5), flat (3-3-5 with regular/expanded/running classification), contracting triangle and contracting diagonal (wave-4 overlap allowed — fixes valid diagonals failing the impulse rules). `RuleResult` gains an additive `isGuideline` flag: failed guidelines flavor a count, only hard rules invalidate
- **Corrective projections.** Forward levels for unfolding B/C waves of zigzags and flats, triangle legs (contraction barrier as invalidation) and the post-triangle thrust; complete corrections project the recovery zone and invalidate beyond wave C
- **Wick-aware, adaptive, multi-scale pivot detection.** The ZigZag detector confirms reversals against candle highs/lows (pivots land on true wick extremes), optionally scales its threshold with volatility (`DetectAtrAdaptive`, k×ATR) and can detect several Elliott degrees at once (`DetectMultiScale`, coarse scales guaranteed subsets of finer ones)
- LLM calls request native JSON mode (`ChatResponseFormat.Json`) where the provider supports it; the robust extraction fallback remains for providers that ignore the hint
- **Google sign-in (end-to-end).** "Continue with Google" on the login screen; the OAuth callback provisions the account just-in-time and issues the same opaque session cookie as password login. Shown only when Google OAuth is configured (`Authentication:Google:ClientId`), surfaced to the frontend via `GET /api/auth/providers`. Post-login redirect is configurable (`Authentication:Google:PostLoginRedirectUri`)
- Deterministic `ElliottRuleChecker`: the three hard Elliott rules + Fibonacci ratios computed in code (objective, no LLM) and returned alongside the AI assessment as `ruleReport`
- The LLM is now grounded on the deterministic checks and prompted as a Socratic **coach** (explanation, alternative count, reflective questions) with a hard guardrail against trading advice
- Frontend shows the objective rule checks and Fibonacci ratios next to the coach reflection
- API serializes enum values as strings

### Security
- Google sign-in now rejects logins whose email Google has not verified (`email_verified`). Without this, an attacker controlling a provider account that asserts someone else's address could take over the matching local account (or pre-provision one). Enforced both at the OAuth callback and inside `ExternalLoginAsync` (defense in depth)
- Per-user rate limiting on the expensive endpoints (`/api/wave-analysis`, `/api/market-data`) to curb LLM-cost / upstream abuse, in addition to the login limiter
- Cap the number of annotations per request (prompt-inflation / DoS guard)
- Stop leaking internal details: generic client errors for upstream/LLM failures (including the raw model output), logged server-side instead
- Honour `X-Forwarded-Proto/For` behind a proxy (correct `Secure` cookie + client IP) and enable HSTS outside Development
- Validate the `days` query range on market data (1–365)
- Stop logging user email addresses; log the non-PII user id instead (CodeQL `cs/exposure-of-sensitive-information`)
- Force the patched `js-yaml` 4.2.0 via npm `overrides` (`npm audit` now reports 0 vulnerabilities)
